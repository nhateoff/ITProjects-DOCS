# Searchable Documentation

## Purpose
Diese Seite erklärt, was **searchable documentation** ist und wie du Dokumentation so schreibst/organisierst, dass man Inhalte über **Suche und Navigation** schnell findet (z. B. GitHub-Suche, Wiki-Suche).

## Definition
Dokumentation ist “searchable”, wenn Inhalte **auffindbar**, **eindeutig benannt** und **gut strukturiert** sind – so, dass Nutzer Antworten schnell finden, ohne lange zu scrollen.

---

## Was macht Doku searchable?

### 1) Scannable Struktur
Wenn eine Doku gut **scannable** ist, wird sie auch besser auffindbar:
- klare Gliederung
- viele sinnvolle Überschriften
- Auflistungen (Bullet/Nummern)

### 2) Saubere Heading-Hierarchie
Eine klare Struktur (H1 → H2 → H3) erleichtert Orientierung und Navigation, besonders bei langen Dokumenten.

### 3) GitHub-Suche gezielt nutzen (Qualifiers)
Mit Such-Qualifiers kannst du Treffer stark eingrenzen:
- `repo:` (Repository)
- `path:` (Ordner)
- `language:` (Dateityp/Format)

Beispiele (anpassbar):
- `repo:nhateoff/ITProjects-DOCS path:docs/guides wiki`
- `repo:nhateoff/ITProjects-DOCS path:docs/concepts pdf`

---

## Checkliste zur Aufarbeitung (Repo-ready)
- [ ] Sprechende Dateinamen (Keywords statt generisch)
- [ ] Einheitliche Heading-Hierarchie (H1/H2/H3)
- [ ] Eine Thematik pro Seite + klare Struktur (Guides/Reference/Troubleshooting)
- [ ] Überschriften so benennen, wie Nutzer wirklich suchen (z. B. “Wikis”, “Pages”, “Knowledge Management”)
- [ ] Startseite/Index mit Quick Links
- [ ] Inhalte so platzieren, dass `path:`-Suche Sinn ergibt
