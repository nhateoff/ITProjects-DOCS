# Wiki Pages (GitHub)

## Purpose
Diese Seite fasst das Thema **Wiki Pages** zusammen: Was ein Wiki ist, wofür es genutzt wird und wie man es auf GitHub sinnvoll strukturiert.

## Definition
**Wikis** sind kollaborative Dokumentationsbereiche, in denen Inhalte schnell erstellt, verlinkt, aktualisiert und versioniert werden. Fokus ist **lebende Dokumentation**: Wissen wird laufend gepflegt statt einmalig publiziert.

Auf **GitHub** ist ein Wiki eine zusätzliche Dokumentationsfläche pro Repository, ideal für **langformige und strukturierte Inhalte** (z. B. Usage-Guides, Entscheidungen, Runbooks).

---

## Typische Merkmale
- Browser-basiertes Bearbeiten (je nach Workflow)
- Interne Verlinkung zwischen Seiten
- Versionshistorie / Nachvollziehbarkeit
- Navigation als Wissensstruktur (z. B. Sidebar)
- Markdown als zentraler Standard (lightweight markup)

---

## Anwendungsbereiche
### Software & IT-Projekte
- Setup, Konfiguration, How-tos, Troubleshooting
- Architektur & Design (Komponenten, Datenmodelle)
- Onboarding (Getting Started, Teamprozesse, Glossar)

### Wissensmanagement in Organisationen
- Wissensdatenbank (Prozesse, Standards, SOPs, Checklisten)
- Qualitätsmanagement (Richtlinien)
- Projektportfolio (Status, Lessons Learned, Templates)

### Bildung / Forschung / Communities
- Kurs-/Lernwikis, gemeinsames Skript
- Forschungsnotizen
- Community-Wissensbasen (FAQ, Contribution-Guides)

---

## Wiki vs. README vs. Pages (Daumenregel)
- **README** = Einstieg + Quickstart + wichtigste Links  
- **Wiki** = Tiefe Inhalte + Betrieb + Troubleshooting + Prozesse  
- **Pages** = öffentliche, “polished” Dokumentations-Website

---

## Kurzer Geschichtsexkurs
- “Wiki” bedeutet “schnell” (Hawaiisch).
- Erstes Wiki: **WikiWikiWeb** (Ward Cunningham, Entwicklung ab 1994, öffentlich seit 1995).
- Motivation: schneller Informationsaustausch und kollaboratives Dokumentieren.

---

## GitHub Wiki: Governance & Workflow
- GitHub Wiki ist **git-basiert** und kann je nach Setup im Browser oder lokal bearbeitet und gepusht werden.
- Typische Governance-Punkte:
  - Wer darf editieren? (Collaborators-only vs. offen)
  - Templates für Seiten (How-to, Concept, Runbook)
  - Pflege: Owner + Review-Zyklus + Archivierung

---

## Ideale Struktur einer Wiki-Page (Best Practice)
1. **Titel** (klar, task-orientiert)  
2. **Purpose** (1–2 Sätze: Ziel)  
3. **Audience** (Dev/Ops/User)  
4. **Prerequisites** (Tools/Rechte/Links)  
5. **Steps / Main Content** (klar, mit Beispielen)  
6. **Verify** (wie prüfe ich Erfolg?)  
7. **Troubleshooting** (Symptom → Ursache → Lösung)  
8. **Security / Notes** (optional)  
9. **Related** (Weiterführende Links)  
10. **Metadata** (Owner, Last reviewed)
